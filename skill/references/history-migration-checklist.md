# 历史信息迁移检查清单

> 当发现重要历史信息仅存在于文件中（备份/会议纪要/项目文档），而未retain到Hindsight时，应按此清单迁移。

## 迁移判断标准

问自己：**这条信息如果未来用户问起，我需要回答吗？**

- 需要回答 → 必须hindsight_retain()到L2
- 不需要回答 → 存文件即可

## 应迁移的内容

| 类别 | 示例 | 重要性 |
|------|------|--------|
| 版本迭代决策 | 为什么选A不选B | ★★★ |
| 用户纠正 | "不要这样做"的原因 | ★★★ |
| 技术踩坑 | 兼容性问题、环境陷阱 | ★★☆ |
| 项目里程碑 | 关键决策和时间节点 | ★★☆ |
| 团队配置演进 | Agent技能分装历史 | ★☆☆ |
| 工具配置决策 | 为什么用四引擎轮询 | ★☆☆ |

## 不应迁移的内容

- 已在MEMORY.md中的LOCKED条目（L1已有）
- 可通过文件名/路径重新发现的信息
- 纯粹的配置值（API密钥、端口号等）
- 已过时且无参考价值的信息

## 迁移操作

```python
# 单条迁移
hindsight_retain(
    content="关键历史信息的内容",
    context="信息来源/背景",
    tags=["类别标签", "时间标签"]
)
```

## 验证迁移成功

```python
# 迁移后立即验证recall能否找到
hindsight_recall(query="关键词")
```

## 典型迁移场景

### 场景1：用户问"之前版本"
1. 先hindsight_recall(query="版本历史")
2. 如果结果不全，检查备份文件是否有遗漏
3. 遗漏的重要信息立即retain
4. 基于recall结果回答

### 场景2：新会话开始
1. 检查是否有新增的备份文件或会议纪要
2. 如有，提取关键决策信息retain到L2
3. 确保下次recall时能找到

### 场景3：用户纠正错误
1. 立即将纠正内容hindsight_retain()
2. 包含：纠正前的错误、纠正后的正确做法、用户原话
3. 确保未来不会重复犯错
