# 记忆迁移与历史查询教训

> 来源：2026-06-19会话中的错误与纠正
> 教训：备份文件和会议纪要必须retain到Hindsight L2

## 一、错误经过

### 错误1：建议不迁移备份到Hindsight
- 用户问"是否将历史备份迁移到Hindsight"
- Agent建议"不迁移"，理由是"备份内容与L1高度重叠"、"防止过时信息干扰"
- 用户随后问"铁壁四规之前版本有哪些"，Agent凭空猜测，驴头不对马嘴

### 错误2：未查历史就回答
- 用户提到"之前版本"，Agent没有先recall Hindsight
- 凭空回答，导致信息错误
- 用户批评："这就是记忆管理的问题...你应该通过Hindsight去检索"

## 二、根因分析

1. **判断错误**：备份文件中的设计决策、技术权衡、用户纠正 = "会犯错"的信息，应该存L2
2. **流程错误**：用户提到历史话题时，必须先hindsight_recall()再回答
3. **分类错误**：备份≠过时信息，很多设计决策有长期价值

## 三、正确做法

### 迁移规则
```
备份文件中的信息分类：
├─ 高价值 → 必须retain到Hindsight L2
│  • 设计决策和理由
│  • 用户纠正和偏好形成过程
│  • 技术踩坑记录
│  • 项目里程碑
│  • 版本迭代主线
├─ 中价值 → 可选迁移
│  • 搜索策略、工具配置
│  • 理论要求
└─ 低价值 → 无需迁移
   • 已在当前L1中的LOCKED条目
   • 已归档到项目文档的详情
   • 可随时查到的API密钥
```

### 查询规则（铁律）
```
用户提到历史话题时：
1. 先 hindsight_recall() 检索L2
2. 再 session_search() 补充L3
3. 最后综合回答
禁止：凭空猜测、跳过recall直接回答
```
