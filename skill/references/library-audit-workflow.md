# 图书馆合规审计工作流

> 适用于：接到项目级任务时，检查图书馆文档的完整性和合规性
> 关联机制：MECH-001（图书馆文档管理）

---

## 触发条件

当以下任一情况发生时，应在主任务开始前执行图书馆审计：
- 收到用户指定的项目问题列表（如"X功能故障、Y展示错误"等）
- 需要创建项目文档时
- 恢复备份后首次操作项目文档

---

## 审计步骤

### 步骤1：核实文件位置合规性

```bash
# 列出项目目录下所有不在日期子目录中的文件
ls ~/.hermes/tushuguan/项目/<项目名>/ | grep -v "^20"
```

**检查项**：
- [ ] 所有文档是否在正确的 `YYYY-MM-DD/` 日期子目录下
- [ ] 项目根目录不应有散落文件（除 `归档/` 目录外）
- [ ] 日期子目录命名格式是否统一（`YYYY-MM-DD`，不是 `YYYYMMDD`）

**修复**：散落文件 → `mv` 移动到最近相关的日期子目录，然后更新 INDEX.md

### 步骤2：核实 INDEX.md 完整性

```
检查 INDEX.md 中日期分组的文档数量，与实际目录中的文件数量对比
```

**检查项**：
- [ ] INDEX.md 中每个日期分组的文档数与目录文件数一致
- [ ] 新文档的行是否被添加（不仅是替换现有行）
- [ ] 变更日志（变更日志表格）是否反映了本次变更
- [ ] 最后更新时间戳是否正确

**修复**：缺失的行 → 补充；数量不匹配 → 找出丢失的索引条目

### 步骤3：核实 INDEX_LOG.md

**检查项**：
- [ ] 最近的文档变更是否记录
- [ ] 日期标注精确到"上午/下午"（同一日期多次变更时）
- [ ] 操作日志与 INDEX.md 变更一致

### 步骤4：创建区间分析文档

当用户报告多个问题时：

```
1. 列出用户报告的所有问题
2. 遍历项目日期目录和 INDEX.md，收集已有方案文档
3. 对照：哪些问题已有方案文档覆盖？哪些没有？
4. 创建补充文档，覆盖没有方案文档的问题
5. 在补充文档顶部注明已有文档覆盖了哪些内容（避免重复）
```

**文档结构建议**：
```markdown
# 项目名-问题诊断补充报告

> 作者：主控Agent
> 日期：YYYY-MM-DD
> 覆盖：未在已有方案文档中记录的问题

---

## 1. 已有文档覆盖情况
| 文档 | 覆盖范围 | 作者 |

## 2. 补充问题（N个，各一章节）
### 问题一：标题
- 根因分析
- 文件/行号
- 严重程度（P0/P1/P2）
- 修复方案

## 3. 修复优先级矩阵
| 优先级 | 问题 | 领域 | 工时 |
```

### 步骤5：串联更新

```
补文件位置 → 更新 INDEX.md → 更新 INDEX_LOG.md
                   ↓
创建补充文档 → 更新 INDEX.md → 更新 INDEX_LOG.md
```

每步操作后立即更新索引，不要等全部完成才更新。

---

## 常见违规模式

| 违规模式 | 表现 | 修复 |
|----------|------|------|
| 文件在根目录 | `项目/AI彩算/视觉优化方案.md` | mv到 `项目/AI彩算/2026-05-30/` |
| INDEX.md漏记 | 目录有文件但INDEX.md无该行 | 补充索引行 |
| 日期格式不一致 | `20260530` vs `2026-05-30` | 统一为 `YYYY-MM-DD` |
| 变更日志缺失 | INDEX.md更新但INDEX_LOG.md为空 | 补充变更日志条目 |
