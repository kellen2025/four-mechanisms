# Recall Before Answering — 历史查询工作流

> 2026-06-19 从用户纠正中总结的核心教训

## 问题

当用户提到历史话题（"之前版本"、"上次我们讨论的"、"那个项目后来怎么样了"）时，
Agent如果不先查L2历史就回答，会导致：
1. 驴头不对马嘴——回答内容与用户所指完全无关
2. 凭空猜测——编造不存在的"历史事实"
3. 用户信任丧失——"你连这个都不记得？"

## 强制流程

```
用户提到历史话题
    ↓
① hindsight_recall(query)  — 检索L2温存储
    ↓ 有结果
基于检索结果回答
    ↓ 无结果
② session_search(query)    — 检索L3冷存储
    ↓ 有结果
基于检索结果回答
    ↓ 无结果
诚实告知："我没有找到相关历史记录"（不要编造）
```

## 关键词触发清单

以下关键词/句式出现时，必须先recall：

| 触发词 | 示例 |
|--------|------|
| 之前/上次/以前 | "之前版本有什么"、"上次讨论的结论" |
| 那个/那个项目 | "那个AI彩算项目后来呢" |
| 还记得 | "还记得我们为什么选方案A吗" |
| 历史/过往 | "历史配置是什么" |
| 变更/改动 | "什么时候改的"、"为什么改了" |
| 版本/迭代 | "v3.0到v3.5改了什么" |
| 决策/选择 | "当时为什么这样决定" |

## L2数据完整性检查

recall查不到的常见原因：
1. **数据未迁移**：备份文件/会议纪要中的历史没有retain到L2
2. **标签不准确**：retain时tags不匹配recall的query
3. **信息过于笼统**：recall query太模糊

解决方案：
1. 定期将重要历史retain到L2（每日20:00自动整理）
2. retain时使用多标签（项目名+日期+主题）
3. recall时尝试不同query角度

## 教训案例

### 案例1：铁壁四规版本历史（2026-06-19）

用户问："3.4.1之前还有什么版本？"

错误做法：
- Agent直接从SKILL.md变更日志中提取（未查L2）
- 结果：给出了版本列表但缺乏决策背景

正确做法：
- 先 hindsight_recall("铁壁四规 版本")
- 检索到：版本迭代主线、各版本核心变更、设计决策理由
- 基于完整历史回答，包含"为什么这样演进"

### 案例2：历史数据迁移决策（2026-06-19）

用户问："是否将历史、备份的记忆迁移到Hindsight中？"

错误做法：
- Agent建议"不迁移"，理由是"备份与现有规则重叠"
- 结果：用户指出这是错误决策，导致历史信息丢失

根本原因：
- Agent没有先recall检查L2中是否有这些历史
- 如果recall发现L2为空，就应该意识到需要迁移
- "不迁移"的建议恰恰证明了Agent不了解L2的现状

正确做法：
- 先 hindsight_recall 检查L2覆盖范围
- 发现L2缺少历史数据 → 建议迁移
- 迁移后验证 recall 能找到历史

## 检查清单

每次会话开始时：
- [ ] 确认Hindsight正在运行（health check）
- [ ] 确认L2中有足够的历史数据
- [ ] 如果L2历史不足，建议用户迁移备份/会议纪要

每次用户提到历史话题时：
- [ ] 是否先recall了？
- [ ] recall结果是否足够回答？
- [ ] 是否需要补充retain新发现的历史？
