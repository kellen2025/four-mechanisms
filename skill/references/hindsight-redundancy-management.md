# Hindsight冗余管理

## 冗余类型

### 1. 完全重复（前100字符相同）

**成因**：同一信息从不同来源（备份文件、会议纪要、会话自动retain）多次写入。

**检测**：recall后对比返回结果的前100字符。

**处理**：保留最早/最完整的版本，删除后续重复。

### 2. 语义重复（内容高度相似）

**成因**：相似信息被从不同角度多次retain（如"铁壁四规版本历史"的多个表述）。

**检测**：recall后人工判断内容是否表达同一事实。

**处理**：保留信息量最大的版本，删除冗余。

### 3. 过时信息

**成因**：项目状态变更后，旧状态未清理（如"API未接入"在接入后仍存在）。

**检测**：定期审计，检查信息是否仍反映当前状态。

**处理**：删除或更新为当前状态。

## 冗余率评估标准

| 冗余率 | 评估 | 建议 |
|--------|------|------|
| <10% | 优秀 | 无需处理 |
| 10-20% | 可接受 | 可选清理 |
| 20-30% | 需关注 | 建议清理 |
| >30% | 过高 | 必须清理 |

## 预防冗余的retain流程

```
准备retain
  ├─ ① 先recall检查是否已存在相似内容
  ├─ ② 如果已存在，判断是否需要更新（而非新增）
  ├─ ③ 如果是新内容，执行retain
  └─ ④ retain后验证：recall确认已存储
```

## 清理操作（谨慎执行）

**清理前**：
1. 备份当前Hindsight数据
2. 列出待清理条目清单
3. 用户确认后执行

**清理后**：
1. 验证关键信息仍可recall
2. 检查冗余率是否降至目标范围
